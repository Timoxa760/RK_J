# MoneyMind Backend — Защита архитектуры

## 1. Почему Go?

**Go, не Python, не Java, не Node.js.**

Go выбран по трём причинам:

- **Одна бинарка** — `go build` даёт исполняемый файл без зависимостей. Для хакатона это критично: мы подняли 15 микросервисов в Docker за 5 минут. В Python нужен виртуальный env со 100+ пакетами, в Node — node_modules на 300 МБ.
- **Горутины** — каждый провайдер чеков (ФНС, X5 Club, Магнит) работает в своей горутине. В Go это встроено в язык, не нужен Asyncio/Twisted/Event Loop.
- **Типизация** — JSON чека проходит через 4 сервиса (scraper → Kafka → receipt-service → Kafka → ai-processor). Если бы не компилятор, расхождение полей мы бы ловили в рантайме на демо.

**Почему не Rust?** Хакатон — скорость разработки важнее скорости выполнения. Go собирается за 2 секунды, Rust — за 2 минуты.

## 2. Почему микросервисы, а не монолит?

**Микросервисы, монолит — выбрали микросервисы.**

MVP можно было сделать монолитом в 3 раза быстрее. Но:

- **Параллельная разработка:** фронтенд пишет `/dashboard/*`, тимлид пишет `/chat`, я пишу `/fns/*`. С монолитом мы бы мержили конфликты в одном `routes.go` каждый час.
- **Независимый деплой:** ai-processor может упасть (ClickHouse не поднят), но scraper-service продолжает качать чеки. В монолите одно падение валит всё.
- **Изоляция данных:** receipt-service работает с PostgreSQL, ai-processor — с ClickHouse. У них разные профили нагрузки. ClickHouse любит батчи по 1000 строк, PostgreSQL — точечные запросы. В одном процессе они конкурируют за ресурсы.

**15 сервисов — много?** Для продакшна — да. Для хакатона — это каркас. 9 из 15 — заглушки (просто `NewRouter()` + `/health`). Мы реализовали только 6 ключевых: `api-gateway`, `user-service`, `receipt-service`, `scraper-service`, `ai-processor`, `analytics-service`. Остальные — "кнопки" под будущие фичи.

## 3. Почему PostgreSQL + ClickHouse, а не одна БД?

**Постгрес — для фактов, КХ — для аналитики.**

Это классическая архитектура **HTAP** (Hybrid Transactional/Analytical Processing), и она здесь оправдана:

**PostgreSQL** — каждая строка чека — это транзакция. Мы делаем `INSERT INTO receipts ... ON CONFLICT DO NOTHING`. Нам нужны:
- Foreign keys (user_id → users)
- `ON CONFLICT` для дедупликации
- JSONB для хранения items в receipts

**ClickHouse** — мы строим агрегаты: `SELECT sum(price*quantity) FROM receipt_items WHERE category = 'Продукты'`. Это analytical workload: сканирование миллионов строк, группировка, материализованные представления.

**Почему не только ClickHouse?** ClickHouse не умеет:
- `ON CONFLICT` для дедупликации
- `RETURNING id` после вставки
- Foreign keys
- Точечные UPDATE — только мутации с накладными расходами

**Почему не только PostgreSQL?** `SELECT category, sum(price*quantity) FROM receipts_items GROUP BY category` — на 10 млн чеков ClickHouse сделает за 50 мс, PostgreSQL — за 3 секунды (даже с индексом). Это подтверждено бенчмарками: ClickHouse в 100-200 раз быстрее на агрегатах.

**Данные дублируются?** Да, сознательно. receipt таблица в PG — источник правды. receipt_items в CH — read-optimized копия для аналитики. Это trade-off: +100 мс на запись, -3 секунды на каждый запрос дашборда.

## 4. Почему Kafka, а не Redis Streams / RabbitMQ / NATS?

**Выбор брокера — Kafka, и вот почему:**

- **Лог с retention** — если ai-processor упал на час, Kafka хранит сообщения (настраиваемый retention). При перезапуске он дочитает с того же места (consumer group + offset). RabbitMQ удалит сообщение после первого ACK. Redis Streams хранит, но не имеет встроенного механизма переигрывания с контролем offset.
- **Партиционирование по ключу** — `receipt.raw` партиционируется по `receipt.ID`. Kafka гарантирует порядок сообщений внутри партиции. Один чек всегда обрабатывается последовательно.
- **Гарантия доставки** — Kafka требует подтверждения от всех in-sync replicas. Мы используем `RequiredAcks: RequireOne`.

**Почему не NATS?** NATS хорош для микросервисного общения (RPC-style), но не умеет хранить историю. Нам нужна воспроизводимость: если мы добавим новое правило категоризации, мы перечитаем `receipt.parsed` с начала (reset consumer offset) и перекатегоризируем все чеки. NATS — at-most-once, Kafka — at-least-once.

**Почему не RabbitMQ?** Rabbit отлично справляется с очередями задач, но плохо — с брокерированием событий. Kafka спроектирован как commit log — то, что нам нужно для event sourcing чеков. RabbitMQ не даёт перечитать старые сообщения без дополнительного кода.

## 5. Почему Chi, а не Gin / Echo / Fiber?

**Chi — самый "стандартный" роутер.**

- **Совместимость с `net/http`** — `chi` реализует стандартный `http.Handler` interface. Gin и Fiber — нет. Это значит, что мы можем использовать любую middleware, написанную под стандартную библиотеку.
- **Встроенный recoverer + CORS** — chi.v5/middleware и chi/cors дают всё нужное из коробки.
- **Нет магии** — Fiber копирует Fasthttp — другой API, другой lifecycle. На хакатоне не время разбираться с нюансами.
- **Контекст** — `chi.RouteContext` передаётся через стандартный `context.Context`, а не через `gin.Context`. Это важно для JWT middleware: мы парсим токен и кладём user_id в контекст — и любой `http.Handler` его читает.

## 6. Каждый сервис — почему существует

### api-gateway (порт 8000)

**Единственная точка входа.** Фронтенд не должен знать, что у нас 15 сервисов. Он стучится на `localhost:8000`, gateway рулит.

**Почему отдельный сервис, а не nginx?** JWT валидация — бизнес-логика. nginx не умеет проверять HMAC-SHA256 JWT без сторонних модулей.

**Почему reverse proxy, а не gRPC?** Фронтенд шлёт JSON. Если бы мы завернули всё в gRPC, пришлось бы ставить grpc-gateway или envoy. На хакатоне это оверхед.

### user-service (порт 8001)

**Регистрация + JWT.** Выделен, чтобы gateway мог доверять токенам. Если бы JWT выпускал scraper-service, это нарушило бы разделение ответственности.

### scraper-service (порт 8003)

**Самый жирный сервис.** Содержит 7 провайдеров: mock, FNS QR, FNS MCO, X5 Club, Magnit, Email IMAP, Email OAuth.

**Почему все провайдеры в одном сервисе?**
- Общая инфраструктура: Kafka producer, rate limiter, credential store
- Общие типы: `RawReceipt`, `RawItem`
- Если бы каждый провайдер был отдельным сервисом — 7 × Dockerfile, 7 × main.go, 7 × портов. Для хакатона это оверхед

**Почему не вынести FNS в отдельный сервис?** FNS — самый ценный источник. Если он упадёт, остальные провайдеры должны продолжать работать. Но на MVP scraper-service — один процесс. В продакшне FNS можно вынести.

### receipt-service (порт 8002)

**Pipeline: валидация → дедупликация → сохранение → отправка на категоризацию.**

**Почему не объединить с scraper-service?**
- Разный lifecycle: scraper может рестартовать для обновления провайдеров, receipt-service — независимо
- receipt-service — единственный, кто пишет в `receipts` таблицу. Единая точка сохранения
- Дедупликация требует БД. scraper-service не должен иметь доступ к PostgreSQL (безопасность)

### ai-processor (порт 8100)

**AI-слой.** Читает `receipt.parsed`, категоризирует (сейчас — словарь, позже — OnlySQ/LLM), пишет в ClickHouse.

**Почему не совместить с receipt-service?**
- ClickHouse — аналитическая БД, PostgreSQL — транзакционная. Они должны жить в разных процессах, чтобы не конкурировать за ресурсы
- Категоризатор — изолируемый модуль. Замена DictCategorizer на OnlySQCategorizer — одна строка в main.go. Если бы он был внутри receipt-service, пришлось бы передеплоивать весь pipeline

**Почему HTTP для manual expenses?** Пользователь вводит расход → ответ должен прийти немедленно. Kafka для этого не подходит (латентность ~50 мс на партицию + обработку).

### analytics-service (порт 8101)

**Заглушка под инсайты/прогнозы.** Выделен, чтобы отделить аналитические отчёты от core pipeline. Если аналитика упадёт, чеки продолжают сохраняться.

### Credit/Budget/Category/Goal/Bank services (порты 8004-8011)

**Заглушки.** Каждая — 20 строк. На MVP не успеваем, но каркас готов: добавь ручку — и сервис работает. Без них пришлось бы втыкать логику в gateway.

## 7. Каждый провайдер — почему выбран

### FNS «Мои Чеки» (mco.nalog.ru) — главный источник

**Почему это primary?** ФНС — государственный сервис с 2018 года собирает все чеки с онлайн-касс. Если чек пробит на кассе — он есть в базе ФНС. Пользователь регистрирует телефон в приложении «Мои Чеки» — и все его чеки автоматически приходят туда.

**Почему Go библиотека `jfk9w-go/lkdr-api`?** Мы не пишем HTTP-клиент с нуля — готовая библиотека с 118 коммитами, которая пережила 53 релиза. Она:
- Сама обновляет токены (refresh token)
- Сама решает капчу (через RuCaptcha или другую имплементацию Authorizer)
- Сама пагинирует чеки

**Почему RuCaptcha?** ФНС поставил reCAPTCHA на страницу входа. RuCaptcha — самый дешёвый (~30 копеек/решение) и стабильный сервис. Альтернатива — 2Captcha (дороже) или AntiCaptcha (та же цена). Решение капчи происходит 1 раз при первой авторизации — токен живёт неделями.

### FNS QR-чеки (proverkacheka.nalog.ru) — fallback

**Зачем, если есть МСО?** Если чек не привязан к телефону (пользователь не регистрировал номер в приложении ФНС) — данные есть только в QR-коде на бумажном чеке. Человек сканирует QR → получает FN, FD, FP → дёргает `proverkacheka.nalog.ru`.

**Почему это бесплатно?** Это открытое API ФНС — не требуется никакой регистрации, ключей, контрактов. Передаёшь FN/FD/FP — получаешь JSON с чеком.

### Mock-провайдер

**Зачем мок, если есть ФНС?** Чтобы разработка не блокировалась отсутствием реальных данных. На хакатоне ClickHouse может быть пуст — mock даёт 2 чека, которые проходят весь pipeline до дашборда. Мы отлаживали полный цикл ещё до того, как подключили FNS.

### X5 Club и Магнит — запасные источники

**Почему реализованы, но не используются?** Было в изначальном плане. X5 Club и Магнит — крупнейшие ритейлеры с программами лояльности. У них есть API для получения истории покупок. Но оба стоят за Cloudflare — без решения капчи и обхода защиты API не работает стабильно.

**Решение:** закопаем моками. Когда появится время + коммерческая необходимость — включим.

## 8. Каждый паттерн — почему так

### Provider Interface

```go
type Provider interface {
    Name() string
    SendCode(ctx context.Context, phone string) error
    Login(ctx context.Context, creds map[string]string) error
    Sync(ctx context.Context) ([]RawReceipt, error)
}
```

**Зачем?** Единый контракт для всех источников чеков. Scheduler не знает, кто он — mock, ФНС или X5 Club. Ему нужен `Provider`. Добавить новый источник — реализовать 4 метода. Не надо менять scheduler, kafka producer, main.go — только добавить провайдер в список.

**Почему SendCode + Login, а не просто Auth(token)?** Разные провайдеры авторизуются по-разному. ФНС МСО — SMS + код. X5 Club — SMS + код. Email — OAuth2. Provider interface должен покрывать любую схему.

### Categorizer Interface

```go
type Categorizer interface {
    Categorize(items []Item) []CategorizedItem
}
```

**Зачем интерфейс для одной функции?** To replace. Сейчас — DictCategorizer (словарь). Завтра — OnlySQCategorizer (LLM). Замена — одна строка:

```go
// Было: cat := categorizer.NewDict()
// Стало: cat := categorizer.NewOnlySQ(onlySQURL)
```

**Почему не LLM сразу?** LLM (даже маленькая) — 200-500 мс на запрос, 50-100 чеков × 10 позиций = 50-100 секунд на синхронизацию. Словарь — 50 мкс на позицию = 5 мс на весь чек. Для MVP скорость важнее точности. LLM подключаем в фазе 2.

### Dedup (SHA-256)

```
Hash = SHA256(receiptID + "|" + provider + "|" + userID)
```

**Почему не по содержимому чека (FN+FD)?** Потому что receiptID уже включает FN+FD (мы формируем `ID = "fns-{FN}-{FD}"`). Дедупликация по ID + провайдер + пользователь — самый надёжный способ: два одинаковых чека от разных провайдеров (FNS MCO и FNS QR) дают одинаковый хеш.

**Почему in-memory cache + DB?** In-memory cache (sync.Map) — это 0-1 мс на проверку. Если бы каждый чек проверялся только через PostgreSQL — 10-50 мс на чек × 500 чеков = 5-25 секунд только на дедуп. Cache живёт 7 дней (столько же, сколько TTL в БД).

**Почему 7 дней?** Чеки синхронизируются раз в 6-24 часа (согласно типу провайдера). 7 дней = запас. Если чек не нашёлся за неделю — его можно повторно добавить без риска дубликата (потерялся в Kafka из-за сбоя).

### ClickHouse Materialized Views

```sql
CREATE MATERIALIZED VIEW spending_by_category
ENGINE = SummingMergeTree()
AS SELECT user_id, category, toStartOfMonth(purchased_at) as month,
   sum(price * quantity) as total
FROM receipt_items
GROUP BY user_id, category, month;
```

**Зачем?** `SELECT sum(price*quantity) FROM receipt_items WHERE user_id='...' AND category='...'` — это full table scan. Материализованное представление считает сумму при вставке, а не при чтении. Дашборд получает ответ за 5-10 мс вместо 1-2 секунд.

**Почему SummingMergeTree?** Таблица может содержать несколько строк с одинаковыми (user_id, category, month) — если чек добавился дважды или пришёл батч. SummingMergeTree схлопнет их при мерже. Нам не нужна абсолютная точность на уровне секунды — отклонение в 0.01% допустимо для графиков.

| Файл | Зачем |
|---|---|

| `001_*.sql` — `008_*.sql` | Версионирование схемы. `up` — создание, `down` — откат. Если бы схема была без миграций — мы бы не знали, какие таблицы есть в продакшне. Каждый SQL — одна атомарная единица изменения |

## 10. Почему DEMO_MODE

**`DEMO_MODE=true`** — архитектурное решение:

- Не требует PostgreSQL — работает с in-memory storage (user-service)
- Не требует ClickHouse — пишет в /dev/null (ai-processor), дашборд возвращает hardcoded JSON
- Не требует Kafka — мок-провайдер не шлёт сообщения, receipt-service не стартует consumer

**Зачем?** Чтобы разработчик `git clone; cp .env.example .env; make run` — и через 5 секунд видит работающий дашборд. Без Docker, без БД, без установки чего-либо. Это ускоряет onboard в 10 раз.

## 11. Почему такие порты

8000-8102 — диапазон выбран не случайно:

- **8000** — gateway (стандартный порт для API-шлюзов)
- **8001-8011** — бизнес-сервисы (замные с основными системами)
- **8100-8102** — AI/аналитика (надстройка над бизнес-слоем)

Это группировка: если порт начинается на 80 — это core, на 81 — это intelligence.

## 12. Что не вошло в MVP и почему

| Что | Почему не вошло |
|---|---|
| **Банковские API** | Условия хакатона (Клерк.Ру): банки не обязательны |
| **X5 Club / Магнит** | Cloudflare — без кастомного решения капчи не работает стабильно |
| **OnlySQ / LLM** | Для MVP скорость (50 мкс/позицию) важнее точности (98% vs 95%) |
| **Real-time уведомления** | WebSocket — отдельный сервис notification-service. Чистое демо можно без них |
| **Система достижений** | gamification — пустой. На демо не показываем |
| **Докеризация всех сервисов** | 9 из 15 — заглушки. Докеризированы для единообразия, но не используются |
